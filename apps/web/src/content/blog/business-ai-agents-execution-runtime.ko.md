AI 에이전트가 90초 동안 "thinking..."만 보여주고 있다면, 사용자는 무엇을 믿어야 할까?

멈춘 것인지, 파일을 읽는 중인지, 외부 API가 실패했는지, 사용자가 방금 보낸 추가 지시가 반영됐는지 알 수 없다. 답변이 아무리 좋아도 이 순간의 제품 경험은 신뢰를 잃는다.

업무용 AI 에이전트의 문제는 더 똑똑한 답변만이 아니다. 사용자가 긴 실행을 맡겼을 때, 그 실행을 관찰하고, 중단하고, 이어가고, 검증할 수 있느냐가 핵심이다.

처음에는 누구나 "얼마나 똑똑한가?"를 본다. 긴 문서를 요약하는지, 코드를 고치는지, 자료를 찾아오는지, 그럴듯한 전략을 쓰는지 본다. 데모에서는 이 질문이 맞다.

하지만 실제 업무에 맡기기 시작하면 질문이 바뀐다.

**"이 에이전트가 지금 무엇을 하고 있는지 믿을 수 있는가?"**

업무용 에이전트는 단순히 좋은 답변을 생성하는 시스템이 아니다. 사용자의 시간, 파일, 지식베이스, 외부 API, 민감한 업무 데이터, 비용이 발생하는 작업과 연결되는 실행 시스템이다. 실행 시스템이라면 모델 성능만으로는 부족하다. 실패를 다루고, 중단을 다루고, 결과물을 남기고, 완료를 검증하고, 사용자가 개입할 수 있게 해야 한다.

우리는 Open Magi Agent를 만들면서 이 문제를 계속 마주했다. 모델이 더 똑똑해지는 것과 별개로, 업무용 에이전트가 신뢰받기 위해서는 모델 주변의 실행 런타임과 UX가 훨씬 단단해야 한다.

업무용 AI 에이전트의 신뢰는 더 똑똑한 답변이 아니라, 중단하고, 재시도하고, 권한을 묻고, 완료를 증명하는 실행 경험에서 나온다.

## 채팅봇의 UX로는 업무용 에이전트를 담을 수 없다

대부분의 AI 제품은 아직 "입력 하나, 출력 하나"의 채팅 UX에서 출발한다.

사용자가 묻는다. 모델이 생각한다. 답변이 나온다.

이 구조는 짧은 질의응답에는 충분하다. 하지만 업무는 그렇게 움직이지 않는다. 실제 업무 요청은 대개 여러 단계로 나뉜다.

- 문서를 읽는다.
- 지식베이스를 검색한다.
- 외부 API를 호출한다.
- 데이터를 비교한다.
- 파일을 만든다.
- 실패한 호출을 다시 시도한다.
- 중간에 사용자가 방향을 바꾼다.
- 사용자 확인이 필요한 작업 앞에서 멈춘다.
- 결과를 검증한 뒤 최종 답변을 작성한다.

이런 작업을 하나의 "thinking..." 상태로 숨겨버리면 사용자는 기다리는 것밖에 할 수 없다. 에이전트가 실제로 일하는 중인지, 멈춘 것인지, 같은 실수를 반복하는지, 사용자 확인을 기다리는지, 이미 답을 만들었는데 화면에 못 보여주는지 알 수 없다.

업무용 에이전트의 UX는 답변 버블이 아니라 **실행 상태**를 보여줘야 한다.

그래서 Open Magi Agent에서는 긴 작업을 단순 로딩 상태로 보지 않는다. 현재 실행 단계, 외부 도구 실행, 하위 작업, 작업 목록, 사용자 확인 대기, 작업 중에 추가로 보낸 지시 같은 상태를 사용자에게 보여줘야 한다고 본다.

이때 중요한 경계가 있다. 모델의 비공개 추론 과정을 보여주자는 뜻이 아니다. 사용자가 알아야 할 것은 모델의 사적인 사고 과정이 아니라, 시스템이 현재 수행 중인 **관찰 가능한 작업 상태**다.

그리고 그 상태는 안전해야 한다. 원본 도구 입력, 민감한 토큰, 내부 프롬프트, 사적인 추론이 그대로 UX 이벤트가 되면 안 된다. 업무용 에이전트의 관찰 가능성은 "많이 보여주기"가 아니라 "사용자가 판단할 수 있는 상태를 안전하게 보여주기"다.

## 중단은 예외가 아니라 대화의 일부다

사람과 일할 때 중간에 말을 끊는 것은 흔한 일이다.

"아, 그 방향 말고 이걸 먼저 해줘."

"잠깐, 그 파일은 건드리지 마."

"방금 생각났는데 이 조건도 넣어줘."

업무용 에이전트도 이 흐름을 받아들여야 한다. 사용자가 에이전트의 긴 작업을 기다리는 동안 새로운 메시지를 보낼 수 있어야 하고, 그 메시지는 버려지거나 다음 세션으로 밀리면 안 된다.

여기서 중요한 것은 stop이나 ESC 같은 중단 동작의 의미다.

일반적인 채팅 UI에서 stop은 "답변 생성을 취소"한다. 하지만 에이전트 UI에서 stop은 더 섬세해야 한다. 현재 생성이나 현재 도구 실행은 멈추되, 사용자가 이미 보낸 추가 지시는 다음 작업으로 이어져야 할 수 있다.

즉, 중단에는 두 가지가 있다.

1. **전체 중단**: 지금 하던 일을 완전히 멈추고 대기 중인 지시도 비운다.
2. **인계형 중단**: 현재 작업을 끊고, 사용자가 작업 중에 보낸 추가 지시를 다음 작업으로 승격한다.

개발자용 에이전트 도구들이 먼저 보여준 중요한 패턴이 있다. 긴 작업 중에도 사용자는 새 지시를 보낼 수 있고, 에이전트는 현재 작업을 적절히 접고 새 맥락을 이어받는다.

Open Magi Agent는 이 패턴을 업무용 에이전트 UX로 가져오고 있다. 사용자가 긴 리서치 중 추가 지시를 보내고 ESC를 누르면, 현재 작업은 중단되고 새 지시가 다음 작업으로 이어져야 한다. 진행 상태와 작업 목록은 사용자가 볼 수 있는 위치에 남아야 한다.

이 차이는 작아 보이지만 중요하다. 사용자의 의도는 대개 "다 그만둬"가 아니라 "지금 하던 건 접고 내가 방금 말한 걸 반영해"에 가깝다. UX는 그 의도를 보존해야 한다.

업무용 에이전트는 사용자가 기다리는 시스템이 아니라, 사용자가 **개입할 수 있는 시스템**이어야 한다.

## 프롬프트 규칙만으로는 신뢰성을 만들 수 없다

많은 에이전트 시스템은 중요한 규칙을 프롬프트에 적는다.

- 실패하면 재시도하라.
- 완료 전에 검증하라.
- 위험한 작업은 사용자에게 물어보라.
- 파일을 만들었으면 전달하라.
- 모르는 것은 모른다고 말하라.

이 규칙들은 맞다. 문제는 프롬프트가 실행 보장을 제공하지 않는다는 점이다.

프롬프트는 모델에게 행동을 요청한다. 하지만 업무용 시스템이 필요한 것은 요청이 아니라 **제어**다.

재시도 가능한 실패는 런타임이 분류하고 복구 경로를 지원해야 한다. 권한 판단은 모델의 선의가 아니라 명시적 권한 경계 위에 있어야 한다. 완료 선언은 좋은 말투가 아니라 검증 증거에 묶여야 한다. 파일 생성은 "경로를 알려주는 문장"이 아니라 사용자가 접근 가능한 산출물 생명주기로 관리되어야 한다.

업무용 에이전트에서 중요한 규칙은 프롬프트 문장이 아니라 1급 컴포넌트가 되어야 한다.

우리는 이것을 실행 런타임이라고 부른다.

실행 런타임은 모델의 지능을 대체하지 않는다. 모델이 업무를 수행할 때 위험한 지점을 시스템적으로 붙잡아준다. 모델은 계획하고 판단하고 작성한다. 런타임은 상태를 기록하고, 권한을 확인하고, 재시도 경로를 제공하고, 중단을 처리하고, 검증을 요구하고, 결과물을 전달한다.

좋은 에이전트는 모델과 런타임이 서로의 역할을 침범하지 않을 때 나온다.

이 원칙들은 추상적인 설계 철학에서 나온 것이 아니다. 긴 작업이 thinking 블록 안에 갇히거나, 사용자의 추가 지시가 현재 작업과 충돌하거나, 작업 계획이 화면에서 멀어져 사용자가 진행 상태를 잃는 문제를 직접 다루면서 만들어졌다. 에이전트 신뢰성은 큰 기능 하나로 해결되지 않았다. 작은 상태 전환들이 정확히 보존될 때 생겼다.

## Open Magi Agent에서 실행 계약은 기본 단위가 된다

업무용 에이전트에 필요한 것은 좋은 의도보다 명확한 기본 실행 단위다. Open Magi Agent에서 우리는 다음과 같은 계약을 제품의 중심에 둔다.

- **실행 타임라인**: 긴 작업을 하나의 로딩 상태가 아니라 관찰 가능한 실행 흐름으로 기록한다.
- **메시지 큐 인계**: 사용자가 중간에 보낸 메시지를 잃지 않고 다음 작업으로 넘긴다.
- **권한 경계**: 위험하거나 민감한 도구 실행은 명시적 확인 흐름을 탄다.
- **작업 목록**: 장기 작업의 계획과 진행 상태를 완료 전까지 사용자가 볼 수 있는 위치에 유지한다.
- **검증 영수증**: 완료 선언은 테스트, 저장, 파일 생성, API 응답 같은 증거와 연결된다.
- **산출물 생명주기**: 결과물은 채팅 텍스트가 아니라 재사용 가능한 업무 산출물로 다룬다.

이 기본 단위들은 화려한 기능처럼 보이지 않을 수 있다. 하지만 긴 업무를 맡기는 순간 차이가 난다.

사용자는 에이전트가 무엇을 하는지 볼 수 있다. 중간에 방향을 바꿔도 맥락이 사라지지 않는다. 실패가 나도 이유와 복구 경로가 남는다. "완료했습니다"가 아니라 "확인했습니다"에 가까워진다. 결과물이 대화 속 문장이 아니라 실제 업무 자산으로 남는다.

이것이 업무용 에이전트가 채팅봇과 달라지는 지점이다.

## 업무용 에이전트에 필요한 여섯 가지 실행 계약

Open Magi Agent를 만들며 우리가 중요하게 보는 계약은 여섯 가지다.

### 1. 작업은 관찰 가능해야 한다

에이전트가 "생각 중"이라고만 표시되면 사용자는 아무것도 판단할 수 없다.

지금 계획 중인지, 검색 중인지, 파일을 읽는 중인지, 하위 작업을 돌리는 중인지, 최종 답변을 작성하는 중인지 구분되어야 한다. 사용자가 시스템을 믿으려면 최소한 현재 작업의 형태를 볼 수 있어야 한다.

관찰 가능성은 개발자용 로그만의 문제가 아니다. 사용자 경험의 문제다.

### 2. 실패는 분류되어야 한다

업무 환경에서 외부 시스템은 항상 실패한다. 모델 API, 지식베이스, 웹 검색, 문서 변환, 파일 업로드, 사내 API 모두 순간적으로 실패할 수 있다.

좋은 에이전트는 모든 실패를 같은 실패로 보지 않는다. 재시도 가능한 네트워크 오류인지, 권한 문제인지, 입력 문제인지, rate limit인지, 사용자가 중단한 것인지 구분해야 한다.

실패를 분류해야 복구 전략이 생긴다. 그렇지 않으면 에이전트는 너무 빨리 포기하거나, 반대로 무의미하게 계속 반복한다.

### 3. 재시도는 런타임 정책이어야 한다

"다시 시도해봐"는 프롬프트에 적을 수 있다. 하지만 어떤 오류에서 몇 번 재시도할지, 얼마나 기다릴지, 어디까지 진행했는지, 같은 작업을 다시 실행해도 안전한지, 사용자에게 무엇을 보여줄지는 런타임이 알아야 한다.

업무용 에이전트의 신뢰성은 거창한 성공보다 작은 실패를 복구하는 능력에서 나온다.

### 4. 권한은 명시적이어야 한다

에이전트가 외부로 데이터를 보내거나, 파일을 바꾸거나, 비용이 드는 API를 호출하거나, 사용자의 업무 상태를 바꾸는 행동을 한다면 권한 경계가 필요하다.

모델이 "이건 괜찮아 보인다"고 판단하는 것만으로는 부족하다. 어떤 행동은 항상 허용되고, 어떤 행동은 항상 차단되고, 어떤 행동은 사용자 확인이 필요해야 한다.

권한은 대화의 분위기가 아니라 시스템 계약이다.

### 5. 완료는 증거 위에 있어야 한다

에이전트가 가장 쉽게 저지르는 실수 중 하나는 너무 빨리 "완료했습니다"라고 말하는 것이다.

코드를 고쳤다면 테스트나 빌드 결과가 있어야 한다. 문서를 만들었다면 파일이 실제로 존재하고 열 수 있어야 한다. 지식베이스에 저장했다면 저장 완료를 확인해야 한다. 외부 API를 호출했다면 응답 상태를 확인해야 한다.

업무에서는 "완료했습니다"보다 "무엇으로 완료를 확인했는가"가 더 중요하다.

### 6. 결과물은 생명주기를 가져야 한다

에이전트가 보고서, 스프레드시트, 문서, 이미지, HTML, 코드, 분석 파일을 만들 수 있다면 그 결과물은 답변의 부속물이 아니다. 업무 산출물이다.

산출물은 최소한 위치와 형식, 접근 방법, 재사용 경로를 가져야 한다. "어딘가에 파일을 만들었으니 알아서 찾아가세요"는 제품 경험이 아니다.

업무용 에이전트는 말로만 끝나는 것이 아니라 결과물을 남겨야 한다.

---

## 상태는 새로고침 후에도 같은 의미를 가져야 한다

실시간 UX만으로는 충분하지 않다.

업무용 에이전트는 긴 작업을 다룬다. 사용자는 브라우저를 새로고침할 수 있고, 네트워크가 끊길 수 있고, 런타임이 재시작될 수 있고, 같은 대화를 나중에 다시 열 수 있다. 이때 메시지 순서가 바뀌거나, thinking 상태에 최종 답변이 갇히거나, 빈 답변 버블이 남거나, 사용자의 추가 지시가 사라지면 신뢰가 깨진다.

따라서 실행 상태는 화면에만 있는 일시적 장식이 아니라 복원 가능한 상태여야 한다.

사용자가 보는 대화 순서, 현재 작업 상태, 대기 중인 지시, 완료된 산출물은 새로고침 후에도 같은 의미를 가져야 한다. 업무용 에이전트의 신뢰성은 실시간 스트리밍 순간뿐 아니라, 다시 돌아왔을 때도 같은 이야기를 하고 있는지에서 결정된다.

---

## 왜 "똑똑한 모델"만으로는 충분하지 않은가

모델은 계속 좋아질 것이다. 더 긴 context window, 더 빠른 추론, 더 좋은 tool use, 더 낮은 가격이 나올 것이다.

그럼에도 업무용 에이전트의 핵심 경쟁력은 모델 호출 한 번에서 끝나지 않는다.

실제 업무에서 사용자가 묻는 질문은 이런 것들이다.

- 방금 에이전트가 왜 멈췄지?
- 내가 중간에 보낸 지시가 반영됐나?
- 이 파일은 어디에 저장됐지?
- 저 작업은 실패했나, 아직 진행 중인가?
- 실패했다면 다시 시도할 수 있나?
- 이 행동은 내 허락을 받고 한 건가?
- 완료됐다는 말은 어떤 검증을 거친 건가?
- 새로고침 후에도 같은 상태를 볼 수 있나?
- 나중에 같은 작업을 다시 추적할 수 있나?

이 질문들에 답하지 못하는 에이전트는 아무리 똑똑해도 업무 시스템이 되기 어렵다.

반대로 이 질문들에 답할 수 있는 에이전트는 완벽하지 않아도 다룰 수 있다. 사용자는 완벽한 동료보다, 현재 상태를 설명하고 실패를 복구하고 필요한 순간에 확인을 요청하는 동료를 더 신뢰한다.

업무용 AI도 같은 기준을 가져야 한다.

---

## Open Magi Agent의 방향

Open Magi Agent는 챗봇을 더 길게 말하게 만드는 제품이 아니다.

우리가 만들고 있는 것은 사용자가 실제 업무를 맡길 수 있는 실행 환경이다. 긴 작업을 관찰할 수 있고, 중간에 개입할 수 있고, 위험한 행동 앞에서는 멈출 수 있고, 실패했을 때 복구할 수 있으며, 완료 후에는 결과물이 남는 에이전트.

앞으로 업무용 AI 에이전트를 평가할 때 우리는 모델 이름만 보지 않을 것이다.

긴 작업을 어디까지 관찰할 수 있는가.

사용자가 중간에 개입해도 상태를 잃지 않는가.

실패를 분류하고 복구할 수 있는가.

권한과 검증이 프롬프트가 아니라 시스템 계약으로 구현되어 있는가.

결과물이 채팅 안에 갇히지 않고 실제 업무 산출물로 남는가.

이 질문에 답할 수 있는 에이전트만이 실제 업무를 맡을 수 있다. Open Magi Agent는 그 기준을 제품의 중심에 두고 있다.

업무용 AI의 다음 단계는 더 빠른 답변이 아니라 더 신뢰할 수 있는 실행이다.

긴 리서치, 문서 작업, 보고서 작성, 지식베이스 기반 업무처럼 "답변"보다 "끝까지 처리되는 실행"이 중요한 일을 AI에게 맡기고 싶다면, Open Magi Agent를 테스트해볼 수 있다.
