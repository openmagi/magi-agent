AIエージェント業界には、誰も語らない不都合な真実があります。

あなたのエージェントは、単にものを忘れるだけではありません。使い続けるほど、**積極的に性能が悪化していく**のです。モデルが劣化するからではなく、contextが劣化するからです。

長期間稼働するAIエージェントが、時間とともに遅くなり、コストが増え、精度が下がっていくのを経験したことがあるなら、まさにこの問題に直面しています。原因はモデルではなく、モデルの*周囲にあるもの*すべてです。

これが**context engineering**の問題です。そして、本番環境のAIエージェント構築において、間違いなく最も重要な未解決課題です。

---

## Contextとは、実際には何なのか？

AIエージェントにメッセージを送るとき、あなたのメッセージだけが入力ではありません。実際の入力は、おおよそ以下のようになっています：

```
[System Prompt]
You are an AI marketing assistant...

[User Profile]
This user runs a small e-commerce business...

[Active Task State]
Currently working on Q1 ad campaign analysis...

[Conversation History]
User: Can you pull the ROAS data for January?
Agent: Here's what I found...
User: Good. Now compare it with December.

[Tool Call Results]
Google Ads API response: { "roas": 3.2, "spend": 12400, ... }
Analytics data: { "sessions": 45200, "conversion_rate": 0.032, ... }

[Current Message]
User: What should we change for February?
```

これらすべてが、毎回のAPI呼び出しで一つの入力としてまとめられます。LLMはすべてを上から下まで読み、レスポンスを生成します。前回の呼び出しの内容を「記憶」しているわけではなく、現在のcontext windowに含まれるものを参照しているだけです。

ここには2つの重大な意味があります：

1. **contextに含まれるものはすべてtokenコストが発生する。** system prompt、会話履歴、ツールの結果。すべてが毎回のAPI呼び出しで課金されます。
2. **contextに含まれるものはすべてattentionを奪い合う。** LLMはすべてのtoken間の関係を同時に計算します（Attentionメカニズム）。無関係な情報が多いほど、attentionは分散します。重要なシグナルがノイズに埋もれてしまうのです。

contextは、エージェントの**コスト**と**品質**の両方を決定します。同時に。追加するすべてのtokenが、プラスにもマイナスにも働きます。

## Context蓄積問題

ここからが厄介です。

多くの人は、contextの蓄積とは「会話が長くなること」だと思っています。それは問題のほんの一部に過ぎません。

実際のシナリオを考えてみましょう。競合他社の価格調査をエージェントに依頼するとします。

その一つの質問に答えるために、エージェントは以下のようなことをするかもしれません：
1. 5社の競合サイトをWeb検索する
2. 価格ページをスクレイピングする（HTMLをmarkdownに変換）
3. 自社の価格履歴に関する社内ドキュメントを読む
4. スプレッドシートからデータを取得する
5. 調査結果を分析してサマリーを作成する

回答が出る頃には、contextに以下が含まれています：
- 5サイト分の競合データ
- 自社の価格ドキュメント
- スプレッドシートデータ
- エージェントの分析と推論
- すべての中間ツール呼び出し結果

これだけで**5万token以上**のリサーチデータがセッションのcontextに残ります。

ここであなたはこう言います。「ありがとう。明日のスタンドアップについてチームにメールを書いてくれる？」

まったく無関係なタスクです。しかし、5万tokenの競合価格リサーチデータは**依然としてcontextに残っています**。課金され続けています。モデルのattentionを奪い続けています。

エージェントは今、競合価格データのことを「考えながら」スタンドアップメールを書いているのです。メールの品質は下がり、コストは倍増し、あなたもエージェントもその理由に気づきません。

**これが根本的な問題です。contextはデフォルトでappend-onlyなのです。** すべてのツール呼び出し、すべての検索結果、すべての中間ステップ。すべてが残り続けます。タスク同士が混ざり合い、コストは複利的に増え、品質は劣化していきます。

そして、ここからさらに悪化します。

## 解決策 #1：Compaction

最も明白な解決策はcompaction（圧縮）です。contextが長くなりすぎたら、LLMに要約させます。

ほとんどのエージェントフレームワークがこれをサポートしています。会話が閾値（例えばcontext windowの80%）に達すると、履歴全体がサマリーに圧縮されます。リセットされ、contextが小さくなります。

エレガントに聞こえますが、実際には2つの致命的な欠陥があります。

### Context Drift

サマリーのサマリーのサマリーは、情報を指数関数的に失います：

- **1回目:** 「ユーザーはReact開発者で、TypeScriptを使ったNext.jsプロジェクトに取り組んでおり、Server Componentsに注力している。」
- **2回目:** 「ユーザーはWeb開発をしている。」
- **3回目:** 「ユーザーはIT業界で働いている。」

たった2〜3回のcompactionサイクルで、重要な詳細が蒸発します。

### 重要度の区別ができない

compactionはすべての情報を平等に扱います。しかし、すべての情報が平等であるわけではありません：

- 「ユーザーは重度のピーナッツアレルギーがある」。命に関わる情報で、数ヶ月後にも必要です。
- 「ユーザーが今日の天気について聞いた」。明日には無関係です。

compactionはこれらを区別できません。すべてに同じ圧縮率を適用します。命に関わる情報が、些細な雑談と一緒に消えてしまいます。

**compactionは、優先度メカニズムのない非可逆圧縮です。** 時間は稼げますが、問題は解決しません。

## 解決策 #2：構造化されたContextファイル

より良いアプローチは、すべてを会話履歴に残すのではなく、重要な情報を構造化されたファイルに書き出すことです。

これは、本格的なエージェント設定で使われる`.md`ベースのcontextパターンです：

- **`MEMORY.md`** 。ユーザーとプロジェクトに関する長期的な事実（約50行）
- **`SCRATCHPAD.md`** 。現在の作業状態とアクティブなタスク（約100行）
- **`AGENTS.md`** 。振る舞いのルールと指示（約500行）

エージェントはセッション開始時にこれらのファイルを読み込みます。compactionで劣化する会話履歴に頼る代わりに、コア情報がセッションをまたいで永続化されるファイルに保存されます。

これは大きな改善です。しかし、新たな問題も生まれます。

**サイズの圧力。** これらのファイルは毎回のAPI呼び出しで読み込まれます。AGENTS.mdの500行は、毎回のメッセージで500行分のtokenが課金されることを意味します。MEMORY.mdを詳細なメモで200行に増やしたら？ ユーザーが「こんにちは」と言っただけでも、毎回200行分のコストがかかります。

**キュレーションの負担。** 誰か（エージェントまたはユーザー）がこれらのファイルに何を入れるか決めなければなりません。多すぎればコスト爆発とattentionの希薄化。少なすぎれば重要な情報が漏れます。

**フラットな構造。** 単一のMEMORY.mdファイルには階層がありません。その情報は昨日のもの？ 先月のもの？ まだ関連がある？ すべてを読まないとわかりません。

構造化ファイルは必要ですが、十分ではありません。「重要な情報はどこにあるか」という問題は解決しますが、「適切な情報を適切なタイミングで見つけるにはどうするか」という問題は解決しません。

## 解決策 #3：RAGの追加

Retrieval-Augmented Generation（RAG）は検索の問題に取り組みます。すべてをcontextに読み込む代わりに、知識を検索可能なインデックスに保存し、関連するものだけを取得します。

エージェントが蓄積した知識をファイルに保存し、検索エンジン（BM25キーワード検索、ベクトル埋め込み、またはその両方）でインデックスを作成します。エージェントが情報を必要とするとき、インデックスを検索して関連するチャンクだけを取得します。

これは強力です。1万件のドキュメント分の知識を持つエージェントでも、各クエリで読み込むのは最も関連性の高い3〜5件だけです。コストは一定。attentionは集中したまま。

しかし、RAGにも限界があります：

**何を検索すべきか事前に知る必要がある。** RAGは明確なクエリがあるときに機能します。しかし、ambient context（エージェントが聞かれなくても「知っておくべき」こと）はどうでしょうか？ ユーザーのタイムゾーン、コミュニケーションの好み、進行中のプロジェクト状況。これらを能動的に検索することはできません。必要だと気づいたときにはもう手遅れだからです。

**インデックスのラグ。** 現在のセッションで書かれた情報は、すぐには検索可能になりません。エージェントは14時に重要なことを学びますが、インデックスが更新されるのはセッション終了後です。その時までに、すでにその情報が必要だったかもしれません。

**時間的な認識がない。** RAGは意味的に最も関連性の高い結果を返しますが、新しさや減衰の概念がありません。3ヶ月前の意思決定と今朝の意思決定が同じ重みで扱われます。実際には、直近のcontextがほぼ常に重要です。

**コールドスタート。** 空の知識ベースを持つ新しいエージェントは何も検索できません。RAGは十分な知識が蓄積された後にしか機能しません。その蓄積には、RAGが解決するはずのcontext管理そのものが必要です。

## 本質的な問題：フルスタックを解決するものがない

各アプローチはそれぞれ一つの側面を解決します：

| アプローチ | 解決すること | 見落としていること |
|----------|--------|--------|
| Compaction | contextのオーバーフロー | 情報損失、優先度なし |
| 構造化ファイル | 永続的なメモリ | スケーリング、キュレーション、フラットな構造 |
| RAG | 検索ベースの取得 | ambient context、時間的認識、コールドスタート |

しかし、本番環境のエージェントには、これらすべてが連携して動作し、さらにその上に何かが必要です。必要なのは以下を満たすシステムです：

1. 生の情報を永続的に保存する（非可逆圧縮なし）
2. 複数の時間スケールで検索可能なインデックスを作成する
3. 適切なcontextを適切なタイミングで読み込む
4. 初日から機能する（コールドスタートなし）
5. 人手によるキュレーションなしに自己メンテナンスする

これが、私たちが構築したものです。

## Compaction Treeの紹介

核心的な洞察は、**オリジナルを決して削除しないこと。その上に検索インデックスを構築すること**です。

図書館に例えるとわかりやすいでしょう。従来のcompactionは、本を燃やして目次だけを残すようなものです。compaction treeは、すべての本を棚に残したまま、カード式目録システムを追加します。

```
memory/
├── ROOT.md                 ← 常に読み込まれる（約100行）
│                              トピックインデックス：「Xについて知っているか？」
├── monthly/
│   └── 2026-03.md          ← 月次キーワードインデックス
│                              「3月のトピック：...」
├── weekly/
│   └── 2026-W11.md         ← 週次サマリー
│                              主要な意思決定、完了タスク
├── daily/
│   └── 2026-03-15.md       ← 日次compactionノード
│                              トピック、意思決定、成果
└── 2026-03-15.md            ← 生の日次ログ（永続、削除しない）
                               起きたことすべての完全な詳細
```

**探索パターン：**

何かを見つけたいとき、上から順にたどります：

1. **ROOT.md** で、トピックインデックスを確認。「競合価格」について知っている？ はい。3月に記録あり。
2. **月次** で、3月のインデックスを見ると、競合分析はW11で実施。
3. **週次** で、W11のサマリーを見ると、価格調査は3月12日。
4. **日次** で、3月12日のノードに主要な意思決定と調査結果あり。
5. **生ログ** で、3月12日の生ログに圧縮されていないオリジナルの全詳細あり。

これは時間的メモリに対する**O(log n)の検索**です。必要以上のものを読むことはありませんが、ドリルダウンすれば常に完全な詳細が利用可能です。

### Fixed（確定）ノード vs Tentative（暫定）ノード

compactionノードにはライフサイクルがあります：

- **Tentative（暫定）** 。その期間がまだ進行中。新しいデータが到着すると再生成されます。今日の日次ノードはtentativeです。今週の週次ノードもtentativeです。
- **Fixed（確定）** 。その期間が終了。ノードは凍結され、二度と更新されません。先週の週次ノードはfixedです。

これにより、ツリーは**初日から使用可能**です。週次サマリーが存在するために1週間待つ必要はありません。tentativeとしてすぐに作成され、新しいデータが到着するたびに更新されます。

### スマートな閾値

すべてにLLMの要約が必要なわけではありません。日次ログが50行であれば、そのまま日次ノードにコピーしてもコストはかからず、情報も失われません。コンテンツが閾値を超えた場合にのみ、LLMによる要約を行います：

| レベル | 閾値 | 閾値以下 | 閾値以上 |
|-------|------|---------|---------|
| 生ログ → 日次 | 約200行 | そのままコピー | LLMによるキーワード密度の高い要約 |
| 日次 → 週次 | 約300行 | 日次ノードを結合 | LLMによる要約 |
| 週次 → 月次 | 約500行 | 週次ノードを結合 | LLMによる要約 |

閾値以下では情報損失ゼロ。閾値以上では、叙述的な読みやすさではなく、検索再現率に最適化されたキーワード密度の高い圧縮を行います。

## Hipocampus：完全なシステム

compaction treeはデータ構造です。[**Hipocampus**](https://github.com/kevin-hs-sohn/hipocampus)は、その周囲に構築された完全なシステムです。私たちが開発し、本番環境で実戦テストし、オープンソース化した3層エージェントメモリプロトコルです。

### 3つのレイヤー

```
Layer 1 — System Prompt（常に読み込み、毎回のAPI呼び出しで使用）
  ├── ROOT.md          約100行   compaction treeのトピックインデックス
  ├── SCRATCHPAD.md    約150行   アクティブな作業状態
  ├── WORKING.md       約100行   現在のタスク
  └── TASK-QUEUE.md    約50行    保留中の項目

Layer 2 — オンデマンド（エージェントが必要と判断した時に読み込み）
  ├── memory/YYYY-MM-DD.md    生の日次ログ（永続）
  ├── knowledge/*.md           詳細なナレッジファイル
  └── plans/*.md               タスクプラン

Layer 3 — 検索（compaction tree + キーワード/ベクトル検索経由）
  ├── memory/daily/            日次compactionノード
  ├── memory/weekly/           週次compactionノード
  └── memory/monthly/          月次compactionノード
```

**Layer 1**は「今何に取り組んでいるか？」に答えます。常にcontextに含まれ、常にコストが発生するため、徹底的に小さく保ちます。

**Layer 2**は「詳細に何を知っているか？」に答えます。アクセスするまでコストは発生せず、エージェントがより多くのcontextが必要だと認識した時にオンデマンドで読み込まれます。

**Layer 3**は「これを以前に見たことがあるか？」に答えます。ROOT.mdのトピックインデックスにより、エージェントは何も読み込むことなく、メモリに情報が存在するかを一目で判断できます。存在する場合、ツリーの探索またはキーワード検索で取得します。

### セッションプロトコル

Hipocampusは2つの必須ルーティンを定義しています：

**セッション開始時：** 何にも応答する前に、エージェントはLayer 1のファイルを読み込み、compactionチェーン（日次 → 週次 → 月次 → Root）を実行します。これにより、ツリーが最新の状態に保たれ、ROOT.mdが最新の状態を反映します。

**タスク完了時のチェックポイント：** タスクを完了するたびに、エージェントは生の日次ファイルに構造化されたログを書き込みます：

```markdown
## Competitor Pricing Analysis
- request: Compare our pricing with top 5 competitors
- analysis: Scraped pricing pages, pulled internal data
- decisions: Recommended 15% reduction on starter tier
- outcome: Report delivered, shared with team
- references: knowledge/pricing-strategy.md
```

これが信頼できる唯一の情報源です。それ以外のすべて（compactionノード、ROOT.md、トピックインデックス）は、compactionチェーンを通じてこれらの生ログから派生したものです。

### ROOT.mdの優位性

最も強力な機能は、ROOT.mdのトピックインデックスです。これは「何を検索すべきか？」という問題を解決します：

```markdown
## Topics Index
- pricing: competitor-analysis, Q1-review, starter-tier-reduction
- infrastructure: k8s-migration, redis-upgrade, node-scaling
- marketing: ad-campaign-Q1, landing-page-redesign, SEO-audit
```

ユーザーが価格について尋ねたとき、エージェントは盲目的に検索する必要はありません。トピックインデックスを確認し、価格情報が存在することを確認し、どの期間にドリルダウンすべきかを正確に把握します。トピックがインデックスにない場合、エージェントは空のメモリを検索して時間を無駄にするのではなく、外部検索を行うべきだと判断できます。

**これにより、「読み込むべきかどうかを判断するために読み込む」という問題が解消されます。** これは、RAGベースのメモリシステムにおける最大の効率性のボトルネックです。

### Proactive Dumps

Hipocampusはタスク完了を待ってcontextを永続化するわけではありません。このプロトコルはproactive dumps（先行的な書き出し）を推奨します。会話が20メッセージ以上続いたとき、重要な意思決定が行われたとき、またはcontextが大きくなっていることをエージェントが感知したときに実行します。

これは、微妙ですが壊滅的な障害モードから保護します。**プラットフォームによるcontextの圧縮**です。ホスティングプラットフォームが会話履歴を圧縮する場合（長時間セッションではほとんどのプラットフォームがそうします）、書き出されていない詳細は永久に失われます。早めに書き出し、頻繁に書き出すこと。生ログはappend-onlyなので、1セッション中に複数回の書き出しを行っても問題ありません。

## なぜこれがエージェントプラットフォームにとって重要なのか

ほとんどのエージェントプラットフォームはデプロイに焦点を当てています。ボタンをクリックすれば、ボットが稼働します。

しかし、デプロイは問題のおそらく5%に過ぎません。残りの95%は**運用**です。エージェントを何週間、何ヶ月にもわたる継続使用で、有用で、正確で、コスト効率よく保ち続けること。

適切なcontext engineeringがなければ：
- エージェントのコストは使用量に比例して増加します
- 無関係な情報がcontextに蓄積され、品質が劣化します
- 重要な知識がcompactionサイクルで失われます
- 昨日知ったことと3ヶ月前に知ったことを区別できません

[Open Magi](https://openmagi.ai)では、自分たち自身が必要だったから[Hipocampus](https://github.com/kevin-hs-sohn/hipocampus)を構築しました。数百のエージェントを本番環境で運用していく中で、すべてのエージェントが同じ壁にぶつかるのを目の当たりにしました。数日間は素晴らしく動くのに、徐々にコストが高く、遅く、忘れっぽくなっていくのです。

Hipocampusは現在、私たちのプラットフォーム上のすべてのエージェントのデフォルトメモリシステムです。Open Magiでエージェントをデプロイすると、APIキー付きのチャットボットだけでなく、context engineeringのフルスタックが手に入ります。階層的compaction、多層メモリ、RAG検索、そして何ヶ月もの継続運用でエージェントをシャープに保つセッションプロトコルです。

エージェントのデプロイは簡単です。*使い続けて役に立つ状態を保つこと*こそが、難しい部分なのです。
